## 1. 需求背景

### 问题

PM Copilot 产出的 PRD、竞品分析、技术规格等 PM 交付物目前以 Markdown 格式显示在对话窗口中。用户需要：

- **分享给非技术干系人**（产品经理、设计师、管理层）— Markdown 不友好
- **归档到文档系统**（Confluence、飞书文档、Notion）— 需要标准格式
- **离线阅读/打印** — 需要 PDF

### 目标用户

| 用户类型 | 使用场景 | 优先级 |
|---------|---------|--------|
| PM 本人 | 导出 PRD/竞品分析供评审会议 | P0 |
| 开发团队 | 接收技术规格文档 | P1 |
| 管理层 | 阅读方案概要/路线图 | P2 |

### 范围

| 交付物 | 导出格式 | 优先级 |
|--------|---------|--------|
| PRD (pm-prd) | PDF + DOCX | P0 |
| 竞品分析 (pm-comp) | PDF | P0 |
| 技术规格 (pm-tech-spec) | PDF + DOCX | P1 |
| 路线图 (pm-roadmap) | PDF | P1 |
| 所有 Skill 输出 | PDF（通用） | P2 |

## 2. 接口设计

### 触发方式

**方案 A：对话内命令**（推荐，与现有 Skill 体系一致）

```
用户: /export [格式] [范围]
示例:
  /export pdf last          # 导出最近一次 Skill 产出为 PDF
  /export docx prd          # 导出最近的 PRD 为 DOCX
  /export pdf all           # 导出当前 session 所有交付物
```

**方案 B：UI 按钮**（需前端变更）

在对话气泡右上角添加"导出"按钮，点击后弹出格式选择。

**决策**：先实现方案 A（零前端变更），方案 B 作为后续增强。

### 导出格式

| 格式 | 用途 | 实现复杂度 |
|------|------|-----------|
| PDF | 通用分享、打印、归档 | 中 |
| DOCX | 可编辑文档、团队协作 | 高 |
| Markdown | 已有（直接复制） | 零 |

### 样式规范

| 元素 | PDF 样式 | 说明 |
|------|---------|------|
| 标题 | 层级分明（H1-H4） | 使用 PM Copilot accent 色 (#ff6b35) |
| 表格 | 交替行背景色 | 边框清晰，表头加粗 |
| 代码块 | 灰色背景 + 等宽字体 | 与对话窗口代码块风格一致 |
| 页眉 | "PM Copilot — {Skill 名称}" | 左侧 |
| 页脚 | 页码 + 生成日期 | 右侧 |
| 水印 | 可选 "Generated by PM Copilot" | 透明度 30% |

## 3. 技术方案对比

### 方案 A: jsPDF + marked（纯前端）

| 维度 | 评估 |
|------|------|
| 原理 | Markdown → HTML (marked) → Canvas (html2canvas) → PDF (jsPDF) |
| 优点 | 零后端依赖，纯浏览器端处理，Tauri WebView 原生支持 |
| 缺点 | 复杂表格渲染差，中文排版有坑，无 DOCX 支持 |
| 包大小 | ~200KB (jsPDF) + ~150KB (html2canvas) |
| 适用 | 简单文档快速导出 |

### 方案 B: Tauri Command + wkhtmltopdf（桌面原生）

| 维度 | 评估 |
|------|------|
| 原理 | Markdown → HTML → wkhtmltopdf 系统命令 → PDF |
| 优点 | 渲染质量高（WebKit 内核），支持复杂布局 |
| 缺点 | 需要系统安装 wkhtmltopdf，跨平台打包复杂，二进制 ~50MB |
| 包大小 | 0（外部依赖）或 ~50MB（内嵌） |
| 适用 | 已有 wkhtmltopdf 环境的专业用户 |

### 方案 C: Tauri Command + Puppeteer/Bun（推荐）

| 维度 | 评估 |
|------|------|
| 原理 | Markdown → HTML (marked) → Puppeteer (headless Chromium) → PDF |
| 优点 | 渲染质量最佳，支持 DOCX (via docx 库)，Bun 已内置 |
| 缺点 | 首次启动 headless browser 稍慢（~2s），需 Node.js 运行时 |
| 包大小 | Puppeteer ~80MB（可选精简版 puppeteer-core ~5MB + 系统 Chrome） |
| 适用 | PM Copilot 已内置 Node.js，可直接使用 |

### 方案 D: Tauri Command + pandoc

| 维度 | 评估 |
|------|------|
| 原理 | Markdown → pandoc → PDF/DOCX |
| 优点 | 格式支持最全（PDF/DOCX/HTML/LaTeX），排版专业 |
| 缺点 | 需要 pandoc 系统安装，PDF 需 LaTeX 引擎，打包困难 |
| 包大小 | pandoc ~30MB + LaTeX ~200MB |
| 适用 | 学术/出版场景 |

### 方案 E: marked + docx 库（轻量推荐）

| 维度 | 评估 |
|------|------|
| 原理 | Markdown → HTML (marked) → PDF (html-pdf-node / Puppeteer-core)；Markdown → AST → DOCX (docx 库) |
| 优点 | 最小依赖，DOCX 原生生成无需外部工具，PM Copilot 已有 Bun/Node |
| 缺点 | PDF 渲染仍需 Chromium，DOCX 复杂表格样式有限 |
| 包大小 | docx ~100KB + marked ~50KB + pdf 库 ~5MB |
| 适用 | PM Copilot 当前架构（推荐起步方案） |

### 推荐决策

| 阶段 | 方案 | 理由 |
|------|------|------|
| **MVP** | 方案 E（marked + docx） | 最小依赖，先覆盖 80% 场景 |
| **V2** | 方案 C（Puppeteer） | 提升 PDF 渲染质量 |
| **不选** | 方案 B/D | 外部依赖过重，与 Tauri 轻量理念冲突 |

## 4. 实现路径

### Phase 1: Markdown → PDF MVP（3 天）

```
输入: Skill 产出的 Markdown 文本
处理: marked → HTML → 注入样式 → Puppeteer-core → PDF
输出: 保存到 ~/Downloads/pm-copilot-{skill}-{date}.pdf
```

关键文件：
- `src/server/export/` — 新增导出模块
- `src/server/export/markdown-to-pdf.ts` — PDF 生成
- `src/server/export/styles.ts` — 样式注入

触发：对话中输入 `/export pdf last`

### Phase 2: Markdown → DOCX（2 天）

```
输入: Skill 产出的 Markdown 文本
处理: marked → AST → docx 库 → DOCX
输出: 保存到 ~/Downloads/pm-copilot-{skill}-{date}.docx
```

关键文件：
- `src/server/export/markdown-to-docx.ts` — DOCX 生成
- `src/server/export/docx-styles.ts` — DOCX 样式

### Phase 3: UI 集成 + 批量导出（2 天）

- 对话气泡添加导出按钮
- 支持 `/export pdf all` 批量导出当前 session 所有交付物
- 导出历史记录

### Phase 4: 质量打磨（1 天）

- 表格跨页处理
- 图片嵌入（如 Mermaid 图表转 PNG）
- 中文字体嵌入（确保 PDF 在无中文字体环境可读）

## 5. 依赖选型

| 依赖 | 版本 | 用途 | License |
|------|------|------|---------|
| marked | ^12.x | Markdown → HTML | MIT |
| docx | ^8.x | DOCX 生成 | MIT |
| puppeteer-core | ^22.x | PDF 渲染 | Apache-2.0 |

**注意**：使用 `puppeteer-core` 而非完整 `puppeteer`，复用系统 Chrome/Edge，避免额外下载 ~300MB 浏览器。

Tauri 环境下 Chromium 查找策略：
1. 优先使用 `TAURI_CHROMIUM_PATH` 环境变量
2. 检测系统 Edge/Chrome 安装路径
3. Fallback 到 Tauri WebView2（仅 Windows）

## 6. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| Puppeteer 在 Tauri 环境无法启动 headless Chrome | 中 | PDF 功能不可用 | Fallback 到 html2canvas + jsPDF（纯前端方案） |
| 复杂 Markdown（嵌套表格、Mermaid）渲染不完整 | 高 | PDF 质量不佳 | MVP 先支持基础排版，复杂元素降级为纯文本 |
| DOCX 中文排版问题 | 中 | 打开后格式错乱 | 使用 docx 库内置 CJK 字体支持 |
| 导出大文档（>50 页）内存占用 | 低 | OOM | 流式处理 + 分页生成 |
| macOS/Linux Chromium 路径差异 | 中 | 跨平台兼容性问题 | 平台特定路径检测 + 用户可配置 |

## 7. 验收标准

| 标准 | 验证方式 |
|------|---------|
| `/export pdf last` 成功生成 PDF | 文件可打开、内容完整 |
| `/export docx last` 成功生成 DOCX | Word/WPS 可打开、可编辑 |
| 表格在 PDF 中正确渲染 | 无截断、无重叠 |
| 中文内容正确显示 | 无乱码、无缺字 |
| 导出文件名含 Skill 名称和日期 | `pm-copilot-prd-2026-05-07.pdf` |
| 导出路径为用户 Downloads 目录 | 跨平台路径正确 |
